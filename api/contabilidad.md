# Contabilidad slice

This document describes how the accounting slice works in this API and how other modules should use it.

## Main purpose

The `Contabilidad` module centralizes:

- accounting years and monthly periods;
- the chart of accounts;
- journal entries and their details;
- reusable journal-entry templates;
- accounting reports such as libro diario, libro mayor, balance de sumas y saldos, balance general, and balance de resultados.

The important rule is that accounting movements must go through complete journal entries. A module should not create loose `AsientosDetalle` rows directly, because the service needs to validate period status, account status, movement type, amount, and double-entry balance.

## Core model

### `ProcesoContable`

Represents an accounting process, normally one fiscal/accounting year.

Important fields:

- `PeriodoAnho`: accounting year.
- `Descripcion`: label for the process.
- `CantNiveles`: maximum chart-of-account depth.
- `CantDigitosNivel`: maximum digits allowed per account level.
- `Moneda`: currency reference.
- `Estado`: controls whether the process can be used.

Enabled states are interpreted by `ContabilidadRules.IsEnabled`. Empty state is treated as enabled. The explicit enabled values are:

- `habilitado`
- `abierto`
- `activo`
- `activa`
- `registrado`
- `registrada`

### `PeriodoContable`

Represents a monthly accounting period inside a `ProcesoContable`.

Important fields:

- `IdProcesoContable`: owning process.
- `Anho` and `Mes`: accounting year/month.
- `FechaInicio` and `FechaFin`: valid date range for journal entries.
- `Estado`: controls whether entries can be created or modified.

`PeriodoContableService` validates:

- `Mes` must be between 1 and 12.
- `FechaInicio` cannot be after `FechaFin`.
- the process must exist and be enabled.
- `Anho` must match `ProcesoContable.PeriodoAnho`.
- `FechaInicio` and `FechaFin` must belong to the indicated month and year.

`PeriodoContableGeneratorService` can create the 12 missing monthly periods for a process through:

```http
POST /api/ProcesosContables/{id}/generar-periodos
```

### `CuentaContable`

`Models/Contabilidad/CuentaContable.cs` is the chart-of-accounts entity.

Important fields:

- `IdProcesoContable`: process/year where the account exists.
- `IdCuentaPadre`: optional parent account.
- `NumeroCuenta`: account code.
- `Nombre`: account name.
- `TipoCuenta`: account category, for example activo, pasivo, patrimonio, ingreso, gasto.
- `EsAsentable`: whether this account can receive journal-entry details.
- `Activa`: whether the account is active.

Relationships:

- one account belongs to one `ProcesoContable`;
- one account can have a parent account and child accounts;
- one account can be used by `AsientosDetalle`;
- one account can be used by `ModelosAsientosDetalle`;
- one account can appear in `BalancesDetalle`.

`CuentaContableService` validates on create/update:

- the referenced `ProcesoContable` must exist;
- the process must be enabled;
- an account cannot be its own parent;
- parent account must exist;
- parent account must belong to the same process;
- account level cannot exceed `ProcesoContable.CantNiveles`;
- `NumeroCuenta.Length` cannot exceed `nivel * CantDigitosNivel`.

Only accounts that are both `Activa = true` and `EsAsentable = true` can be used in journal-entry details.

### `Asiento`

Represents the journal-entry header.

Important fields:

- `NumeroAsiento`: journal-entry number inside the period.
- `IdPeriodoContable`: accounting period.
- `IdModulo`: optional source module.
- `Fecha`: accounting date.
- `Descripcion`: entry description.
- `Automatico`: whether the entry was generated by a module/template.
- `Estado`: current entry state.
- `ReferenciaOrigen`: text reference to the source document/module.
- `IdOrigen`: source record id.
- `FechaMayorizacion`: optional posting date.

Direct create/update through normal CRUD is blocked by `AsientoService`. Use:

```http
POST /api/Asientos/completo
PUT /api/Asientos/completo/{id}
POST /api/Asientos/generar-desde-modelo
```

### `AsientosDetalle`

Represents each debit/credit line of an `Asiento`.

Important fields:

- `IdAsiento`: parent journal entry.
- `IdCuentaContable`: account to affect.
- `Item`: line order.
- `TipoMovimiento`: must be `Debe` or `Haber`.
- `Monto`: must be greater than zero.
- `DescripcionItem`: optional line description.

Direct create/update/delete is blocked by `AsientosDetalleService`. Details are replaced as a complete set when updating an `Asiento` through `/api/Asientos/completo/{id}`.

### `ModeloAsiento` and `ModelosAsientosDetalle`

Templates used to generate automatic journal entries.

`ModeloAsiento` defines:

- source `IdModulo`;
- `TipoAsiento`;
- description and summary;
- active/inactive state.

`ModelosAsientosDetalle` defines template lines:

- account;
- item/order;
- movement type (`Debe` or `Haber`);
- default line description.

When generating an entry from a model, the request provides the amount for each template detail. The service copies the accounts and movement types from the model and creates a complete `Asiento`.

## Main services

### `CuentaContableService`

Handles the chart of accounts and includes the hierarchy/process validations described above.

Main API controller:

```http
/api/CuentasContables
```

### `AsientoContableService`

This is the main service that other modules should use for accounting movements.

It supports:

- manual complete entry creation;
- complete entry update;
- generation from an active `ModeloAsiento`.

For every create/update it validates:

- an enabled period exists for the entry date;
- period and process are enabled;
- there is at least one detail;
- every detail movement is `Debe` or `Haber`;
- every amount is greater than zero;
- total Debe equals total Haber;
- each account belongs to the period's process;
- each account is active and asentable.

The client does not send `IdPeriodoContable` when creating a complete asiento. The backend resolves the period from `Fecha`, using the enabled period whose date range contains that date. If `NumeroAsiento <= 0`, the service assigns the next number in the resolved period.

Create/update runs inside a database transaction. On update, existing details are deleted and recreated from the submitted list.

### `ContabilidadRules`

Shared validation helper for:

- enabled state checks;
- `Debe`/`Haber` normalization;
- movement validation;
- period validation;
- date-inside-period validation;
- account-is-asentable validation;
- double-entry validation.

### `ContabilidadReportesService`

Builds reports from `AsientosDetalle`, including the related `Asiento` and `CuentaContable`.

Available endpoints:

```http
GET /api/contabilidad/reportes/libro-diario/{idPeriodoContable}
GET /api/contabilidad/reportes/libro-mayor/{idPeriodoContable}
GET /api/contabilidad/reportes/balance-sumas-saldos/{idPeriodoContable}
GET /api/contabilidad/reportes/balance-general/{idPeriodoContable}
GET /api/contabilidad/reportes/balance-resultados/{idPeriodoContable}
```

`Libro Diario` returns ordered movement lines by date, entry number, and item.

`Libro Mayor` groups diario lines by account and calculates:

- total Debe;
- total Haber;
- saldo deudor;
- saldo acreedor;
- movements for each account.

`Balance de Sumas y Saldos` groups movements by account and calculates totals and saldo.

`Balance General` filters balance lines where `TipoCuenta` contains activo, pasivo, patrimonio, or capital.

`Balance de Resultados` filters balance lines where `TipoCuenta` contains ingreso, venta, gasto, egreso, costo, or resultado.

## Standard flow

### 1. Create or enable the accounting process

Create a `ProcesoContable` for the accounting year.

Example intent:

```json
{
  "periodoAnho": 2026,
  "descripcion": "Contabilidad 2026",
  "cantNiveles": 5,
  "cantDigitosNivel": 2,
  "moneda": "PYG",
  "estado": "Habilitado"
}
```

### 2. Generate or create periods

Use:

```http
POST /api/ProcesosContables/{idProcesoContable}/generar-periodos
```

This creates missing periods from January to December using the process year.

### 3. Create the chart of accounts

Use:

```http
POST /api/CuentasContables
```

Create parent/group accounts with `EsAsentable = false` and movement accounts with `EsAsentable = true`.

Example structure:

```text
1 Activo                 EsAsentable=false
1.01 Caja y Bancos       EsAsentable=false
1.01.01 Caja             EsAsentable=true
2 Pasivo                 EsAsentable=false
2.01 Proveedores         EsAsentable=true
5 Gastos                 EsAsentable=false
5.01 Compras             EsAsentable=true
```

### 4. Optionally create journal-entry models

Use `ModelosAsientos` and `ModelosAsientosDetalles` for recurring entries that modules can reuse.

Example: a purchases module can have a model with:

- Debe: purchases/expense account.
- Haber: supplier/payable account.

The module later provides only date, source reference, source id, and amounts for the model detail lines.

### 5. Create accounting entries

Manual entry:

```http
POST /api/Asientos/completo
```

Example:

```json
{
  "numeroAsiento": 0,
  "idModulo": 2,
  "fecha": "2026-03-15",
  "descripcion": "Registro factura de compra",
  "automatico": true,
  "estado": "Registrado",
  "referenciaOrigen": "FacturasCompra",
  "idOrigen": 120,
  "detalles": [
    {
      "idCuentaContable": 10,
      "item": 1,
      "tipoMovimiento": "Debe",
      "monto": 100000,
      "descripcionItem": "Mercaderias"
    },
    {
      "idCuentaContable": 25,
      "item": 2,
      "tipoMovimiento": "Haber",
      "monto": 100000,
      "descripcionItem": "Proveedor"
    }
  ]
}
```

Template-based entry:

```http
POST /api/Asientos/generar-desde-modelo
```

Example:

```json
{
  "idModeloAsiento": 4,
  "numeroAsiento": 0,
  "fecha": "2026-03-15",
  "descripcion": "Factura de compra 001-001-0000120",
  "referenciaOrigen": "FacturasCompra",
  "idOrigen": 120,
  "detalles": [
    {
      "idModeloAsientoDetalle": 8,
      "monto": 100000,
      "descripcionItem": "Mercaderias"
    },
    {
      "idModeloAsientoDetalle": 9,
      "monto": 100000,
      "descripcionItem": "Proveedor"
    }
  ]
}
```

## How another module should use Contabilidad

For a source module such as Compras or Ventas:

1. Complete the business transaction in its own tables.
2. Send the transaction `Fecha`; Contabilidad resolves the enabled `PeriodoContable`.
3. Resolve the needed `CuentaContable` ids or choose a `ModeloAsiento`.
4. Call `IAsientoContableService.CreateManualAsync` or `GenerateFromModeloAsync`.
5. Send `ReferenciaOrigen` with the source table/module name.
6. Send `IdOrigen` with the source record id.
7. Set `Automatico = true` for module-generated entries.

This gives traceability from accounting back to the source document:

```text
FacturasCompra.IdFacturaCompra -> Asiento.ReferenciaOrigen = "FacturasCompra"
FacturasCompra.IdFacturaCompra -> Asiento.IdOrigen
```

If the source transaction is edited and accounting must change, call:

```http
PUT /api/Asientos/completo/{idAsiento}
```

The service replaces all details atomically and revalidates double entry.

## Important constraints

- Do not use `/api/Asientos` POST/PUT for normal entry creation. The service blocks it.
- Do not use `/api/AsientosDetalles` POST/PUT/DELETE directly. The service blocks it.
- Use only active, asentable accounts in entry details.
- Accounts used by an entry must belong to the same process as the period resolved from `Fecha`.
- Entry date must match exactly one enabled period.
- Debe and Haber totals must match exactly.
- Updating a complete entry replaces all details.
- Deleting an `Asiento` is allowed only if its period and process are enabled.

## Short mental model

```text
ProcesoContable
  -> PeriodoContable
  -> CuentaContable
  -> Asiento
      -> AsientosDetalle
          -> CuentaContable

ModeloAsiento
  -> ModelosAsientosDetalle
  -> Asiento generated by AsientoContableService

AsientosDetalle
  -> ContabilidadReportesService
  -> Libro Diario / Mayor / Balances
```

The module is designed so that `CuentaContable` defines where money can move, `PeriodoContable` defines when it can move, and `AsientoContableService` is the gatekeeper that validates every movement before it reaches the database.
